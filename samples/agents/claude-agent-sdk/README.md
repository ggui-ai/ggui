# Sample Agent — Claude Agent SDK ↔ ggui

Reference backend for an MCP-host agent built with the Anthropic Claude Agent SDK (`@anthropic-ai/claude-agent-sdk`), pointed at a ggui MCP server.

**The agent core is a small, readable amount of TypeScript.** No ggui agent-side wrapper, no `defineAgent` helper, no scaffold. The only ggui-specific configuration is the MCP server URL — every ggui tool (`ggui_handshake`, `ggui_render`, `ggui_update`, `ggui_emit`, `ggui_consume`) is discovered by Claude via the standard MCP `tools/list` handshake.

This is "Zero Agent Code" made concrete.

## Architecture

This package is the **backend** only. There is no bundled frontend — the reference frontend lives at `oss/samples/apps/ggui-basic-web/` (a Vite SPA that mounts `<AppRenderer>` from `@ggui-ai/react`) and binds to this backend via `VITE_AGENT_ENDPOINT_URL`.

```
src/
  agent.ts    runAgent({ prompt, mcpServers, chatId, … }) — SDK-specific loop.
              Translates the brand-agnostic `mcpServers` map into the Claude
              Agent SDK's native `{type:'http', url, headers}` shape and
              yields normalized SDKMessage events.
  server.ts   Pure-Node HTTP API (no framework). Exposes /chat (SSE),
              /chat?chatId= snapshot, /relay/tools-call,
              /relay/resources-read, /api/renders/:id/state, and
              /sandbox-proxy-url. Brand-agnostic — knows nothing about
              ggui's _meta envelope.
  index.ts    Boots the server with env-derived options. The only place
              that maps env vars to the mcpServers config map.
```

Three layers, one direction. Each file's docstring spells out its boundary and what it deliberately doesn't do.

## Wire diagram

```
┌─────────────────────────────────────┐
│ Browser — ggui-basic-web (Vite)     │
│ <AppRenderer> from @ggui-ai/react   │
└─────────┬───────────────────┬───────┘
          │                   │
          │ POST /chat        │ iframe-internal
          │ (SSE stream of    │ tools/call + resources/read
          │  SDKMessage)      │ (postMessage to host)
          ▼                   ▼
┌─────────────────────────────────────┐
│ server.ts (pure node:http)          │
│   /chat ──► runAgent(prompt, …)     │
│   /relay/tools-call    ──► ggui MCP │
│   /relay/resources-read ──► ggui MCP │
│   /api/renders/:id/state ──► ggui MCP│
└─────────┬───────────────────────────┘
          │
          │ for await msg of runAgent(…)
          ▼
┌─────────────────────────────────────┐
│ agent.ts                            │
│ query({                             │
│   prompt,                           │
│   mcpServers: { ggui: {type:'http', │
│                       url, headers}}│
│   allowedTools: ['mcp__ggui__…']    │
│ })                                  │
└─────────┬───────────────────────────┘
          │ MCP JSON-RPC
          ▼
┌─────────────────────────────────────┐
│ ggui MCP server                     │
│ (`ggui serve --port 6781 --mcp-only`)│
└─────────────────────────────────────┘
```

The iframe content the user sees is generated by ggui and exposed as a spec-canonical MCP resource at `ui://ggui/render/<renderId>`. The host fetches that resource via standard MCP `resources/read` (proxied through `/relay/resources-read` so the browser holds no MCP credential) — same code path every MCP-Apps-spec host uses.

Inside the iframe, user actions (button clicks, form submits) are routed as standard `tools/call:ggui_runtime_submit_action` postMessage notifications. The host frontend forwards them through `/relay/tools-call` to the ggui MCP, which wakes the agent's blocked `ggui_consume` call so the LLM can react.

The runtime bridge that turns those postMessage calls into App-class methods is `@modelcontextprotocol/ext-apps`, the spec-canonical client adopted in Phase 1.19b.3.

## Running standalone

```bash
# 1. Boot a ggui MCP server in another terminal
ggui serve --port 6781 --mcp-only

# 2. Set your Anthropic API key
export ANTHROPIC_API_KEY=sk-ant-...

# 3. Boot the sample agent backend
pnpm --filter @ggui-samples/agent-claude-sdk start
# → backend ready on http://localhost:6790
# → sandbox proxy on http://localhost:7790 (spec-mandated second origin)

# 4. Boot the reference frontend in another terminal
VITE_AGENT_ENDPOINT_URL=http://localhost:6790 \
  pnpm --filter @ggui-samples/app-ggui-basic-web dev
# → frontend on http://localhost:6890
```

### Env vars

| Var                  | Default                     | Purpose                                                   |
| -------------------- | --------------------------- | --------------------------------------------------------- |
| `PORT`               | `6790`                      | Chat backend HTTP port                                    |
| `SANDBOX_PROXY_PORT` | `7790`                      | Spec-mandated second-origin sandbox port                  |
| `GGUI_MCP_URL`       | `http://localhost:6781/mcp` | Primary ggui MCP endpoint                                 |
| `GGUI_TODO_MCP_URL`  | —                           | Optional second MCP for domain tools (todo demo)          |
| `GGUI_MCP_BEARER`    | `dev`                       | Sent as `Authorization: Bearer <token>` on every MCP req  |
| `MODEL`              | `claude-haiku-4-5`          | Claude model id                                           |
| `SYSTEM_PROMPT`      | (canonical posture prompt)  | Override the default; set `none` to disable               |
| `ANTHROPIC_API_KEY`  | —                           | Required; the agent fails-fast on first `/chat` if absent |

## Adding another MCP server

Two diffs in this sample to wire a third MCP:

1. **`src/index.ts`** — add an entry to the `mcpServers` map (and an env var to drive it):

   ```ts
   if (process.env.GGUI_WEATHER_MCP_URL) {
     mcpServers.weather = { url: process.env.GGUI_WEATHER_MCP_URL };
   }
   ```

2. **`src/agent.ts`** — add an entry to `ALLOWED_TOOLS_BY_SERVER` so the LLM is allowed to call its tools:

   ```ts
   weather: ['mcp__weather__weather_current', 'mcp__weather__weather_forecast'],
   ```

That's it. The map key (`weather`) becomes the MCP namespace; tools get auto-prefixed as `mcp__weather__<tool>`. The OpenAI and Google ADK samples use the same map shape but don't need step 2 — their SDKs use the configured-servers tool catalog directly.

## Swapping SDKs

This sample is one of three SDK-specific backends with identical wire surface area:

- `@ggui-samples/agent-claude-sdk` (this) — Claude
- `@ggui-samples/agent-openai-sdk` — OpenAI Agents SDK (GPT-5.5)
- `@ggui-samples/agent-google-adk` — Google ADK (Gemini)

To swap drivers, point the frontend's `VITE_AGENT_ENDPOINT_URL` at the sample backend you're running — all three samples now default to 6790, so override `PORT` to run more than one at once. The same `<AppRenderer>` works against any of them because the wire shape (POST /chat SSE of `SDKMessage`-shaped events, ggui MCP `_meta` envelope) is identical.

## Plugging into a non-Vite frontend

The backend is pure node:http with wide-open CORS for dev — any frontend that can fetch SSE, fetch JSON, and mount `<AppRenderer>` (or any MCP-Apps-spec iframe host) works:

- **Next.js** — fetch `/chat` from a route handler or directly from a client component
- **Remix / Astro / SvelteKit** — same; the wire is HTTP+SSE
- **Plain HTML** — `EventSource('http://localhost:6790/chat')` works

The frontend needs to:

1. POST a prompt to `/chat` and read the SSE stream
2. Fetch `/sandbox-proxy-url` once on mount and thread the URL into `<AppRenderer sandbox.url>`
3. Implement `onToolCall` / `onReadResource` to forward iframe requests to `/relay/tools-call` and `/relay/resources-read`

The `useMcpAppsChat` hook in `@ggui-ai/react/chat-helpers` does all three.

## Used by

The end-to-end suite spawns this sample as the agent side of its test scenarios, pairing it with the `samples/gguis/*` operator configurations (`default`, `canvas-demo`, `leaflet-demo`, `mapbox-demo`).

## Not used for

Not a published library. Not a scaffold. Not a framework. This file exists to be **read** by external developers building their own agents — the patterns here transfer directly to any MCP-host code path.
