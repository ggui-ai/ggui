# Sample Agent — Claude Agent SDK ↔ ggui

Reference backend for an MCP-host agent built with the Anthropic Claude Agent SDK (`@anthropic-ai/claude-agent-sdk`), pointed at a ggui MCP server.

**The agent core is a small, readable amount of TypeScript.** No ggui agent-side wrapper, no `defineAgent` helper, no scaffold. The only ggui-specific configuration is the MCP server URL — every ggui tool (`ggui_handshake`, `ggui_render`, `ggui_update`, `ggui_emit`, `ggui_consume`) is discovered by Claude via the standard MCP `tools/list` handshake.

This is "Zero Agent Code" made concrete.

## Architecture

This package is the **backend** only. There is no bundled frontend — the reference frontend lives at `oss/samples/apps/ggui-basic-web/` (a Vite SPA that mounts `<AppRenderer>` from `@mcp-ui/client`, the spec-canonical MCP Apps host, with ggui chat plumbing from `@ggui-ai/react/chat-helpers` — ggui deliberately does not wrap or re-export the host) and binds to this backend via `VITE_AGENT_ENDPOINT_URL`.

```
src/
  agent.ts    createClaudeAgentSdkAdapter(…) — the SDK-specific piece.
              Translates the brand-agnostic `mcpServers` map into the Claude
              Agent SDK's native `{type:'http', url, headers}` shape and
              yields normalized SDK messages.
  server.ts   Thin glue: wires the adapter into `startAgentServer` from
              `@ggui-ai/agent-server`. The library (a Hono app) owns the
              whole HTTP surface — GET `/` manifest, GET+POST `/agent`,
              `/auth/*` — plus MCP routing, auth, chat ownership, and
              tool-result resource inlining. Brand-agnostic — knows
              nothing about ggui's _meta envelope.
  index.ts    Boots the server with env-derived options. The only place
              that maps env vars to the mcpServers config map.
```

Three layers, one direction. Each file's docstring spells out its boundary and what it deliberately doesn't do.

## Wire diagram

```
┌─────────────────────────────────────┐
│ Browser — ggui-basic-web (Vite)     │
│ <AppRenderer> from @mcp-ui/client   │
│ + useMcpAppsChat (@ggui-ai/react)   │
└─────────┬───────────────────┬───────┘
          │                   │
          │ POST /agent       │ iframe-internal
          │ {kind:'chat'}     │ tools/call
          │ (SSE stream)      │ (postMessage to host)
          ▼                   ▼
┌──────────────────────────────────────┐
│ @ggui-ai/agent-server (Hono)         │
│   GET  /            ──► manifest     │
│   GET  /agent?chatId= ──► snapshot   │
│   POST /agent {kind:'chat'}          │
│        ──► adapter.run(…) SSE        │
│   POST /agent {kind:'tool-call'}     │
│        ──► relayed to the ggui MCP   │
│   /auth/* ──► guest-token adapter    │
└─────────┬────────────────────────────┘
          │ for await msg of adapter.run(…)
          ▼
┌─────────────────────────────────────┐
│ agent.ts                            │
│ query({                             │
│   prompt,                           │
│   mcpServers: { ggui: {type:'http', │
│                       url, headers}}│
│   allowedTools: ['mcp__ggui__…']    │
│ })                                  │
└─────────┬───────────────────────────┘
          │ MCP JSON-RPC
          ▼
┌─────────────────────────────────────┐
│ ggui MCP server                     │
│ (`ggui serve --port 6781 --mcp-only`)│
└─────────────────────────────────────┘
```

The iframe content the user sees is generated by ggui and exposed as a spec-canonical MCP resource at `ui://ggui/render/<renderId>`. The agent-server library fetches that resource via standard MCP `resources/read` and inlines it into the tool result before it reaches the browser — the browser holds no MCP credential. On rehydration (`GET /agent?chatId=`), each recorded tool-result's resource is re-inlined fresh, so a reloaded tab reflects the current server-authoritative render state.

Inside the iframe, user actions (button clicks, form submits) are routed as standard `tools/call:ggui_runtime_submit_action` postMessage notifications. The host frontend forwards them as `POST /agent {kind:'tool-call'}`; the library relays the call to the ggui MCP, which wakes the agent's blocked `ggui_consume` call so the LLM can react.

The runtime bridge that turns those postMessage calls into App-class methods is `@modelcontextprotocol/ext-apps`, the spec-canonical client.

## Running standalone

```bash
# 1. Boot a ggui MCP server in another terminal
ggui serve --port 6781 --mcp-only

# 2. Set your Anthropic API key
export ANTHROPIC_API_KEY=sk-ant-...

# 3. Boot the sample agent backend
pnpm --filter @ggui-samples/agent-claude-sdk start
# → backend ready on http://localhost:6790
# → sandbox proxy on http://localhost:7790 (spec-mandated second origin)

# 4. Boot the reference frontend in another terminal
VITE_AGENT_ENDPOINT_URL=http://localhost:6790 \
  pnpm --filter @ggui-samples/app-ggui-basic-web dev
# → frontend on http://localhost:6890
```

### Env vars

| Var                  | Default                     | Purpose                                                  |
| -------------------- | --------------------------- | -------------------------------------------------------- |
| `PORT`               | `6790`                      | Chat backend HTTP port                                   |
| `SANDBOX_PROXY_PORT` | `7790`                      | Spec-mandated second-origin sandbox port                 |
| `GGUI_MCP_URL`       | `http://localhost:6781/mcp` | Primary ggui MCP endpoint                                |
| `GGUI_TODO_MCP_URL`  | —                           | Optional second MCP for domain tools (todo demo)         |
| `GGUI_MCP_BEARER`    | `dev`                       | Sent as `Authorization: Bearer <token>` on every MCP req |
| `MODEL`              | `claude-sonnet-4-6`         | Claude model id                                          |
| `SYSTEM_PROMPT`      | (canonical posture prompt)  | Override the default; set `none` to disable              |
| `ANTHROPIC_API_KEY`  | —                           | Required; the agent fails fast at boot if absent         |

## Adding another MCP server

One env var — `src/index.ts` scans the environment for `GGUI_<NAME>_MCP_URL` and auto-registers each match as the `<name>` MCP namespace, so wiring a third MCP needs **no `index.ts` change**:

```bash
GGUI_WEATHER_MCP_URL=http://localhost:6783/mcp
```

**Claude SDK only — one extra diff:** the Claude Agent SDK's `allowedTools` matches EXACT tool names (there is no server-wildcard), so also add the new server's tools to `DEFAULT_ALLOWED_TOOLS` in `src/agent.ts` (or pass the `allowedToolsByServer` option to `createClaudeAgentSdkAdapter`):

```ts
weather: ['mcp__weather__weather_current', 'mcp__weather__weather_forecast'],
```

The env-var name (`weather`) becomes the MCP namespace; tools get auto-prefixed as `mcp__weather__<tool>`. The OpenAI and Google ADK samples use the same env scan but don't need the allowlist step — their SDKs use the configured-servers tool catalog directly.

## Swapping SDKs

This sample is one of three SDK-specific backends with identical wire surface area:

- `@ggui-samples/agent-claude-sdk` (this) — Claude
- `@ggui-samples/agent-openai-sdk` — OpenAI Agents SDK (GPT-5.5)
- `@ggui-samples/agent-google-adk` — Google ADK (Gemini)

To swap drivers, point the frontend's `VITE_AGENT_ENDPOINT_URL` at the sample backend you're running — all three samples now default to 6790, so override `PORT` to run more than one at once. The same `<AppRenderer>` works against any of them because the wire shape (POST `/agent` SSE of normalized SDK events, ggui MCP `_meta` envelope) is identical.

## Plugging into a non-Vite frontend

The backend is a small Hono app with wide-open CORS for dev — any frontend that can fetch SSE, fetch JSON, and mount `<AppRenderer>` (or any MCP-Apps-spec iframe host) works:

- **Next.js** — fetch `/agent` from a route handler or directly from a client component
- **Remix / Astro / SvelteKit** — same; the wire is HTTP+SSE
- **Plain HTML/JS** — `fetch('http://localhost:6790/agent', { method: 'POST', … })` + ReadableStream parsing works (the chat stream is SSE over POST, so `EventSource` — GET-only — does not apply)

The frontend needs to:

1. `GET /` once on mount and thread the manifest's `sandboxProxyUrl` into `<AppRenderer sandbox.url>`
2. POST `{kind:'chat', prompt, chatId?}` to `/agent` and read the SSE stream (the first event is `chat-allocated` with the server-assigned chat id; rehydrate later via `GET /agent?chatId=`)
3. Forward iframe `tools/call` requests as `POST /agent {kind:'tool-call', name, arguments}`

The `useMcpAppsChat` hook in `@ggui-ai/react/chat-helpers` does all three.

## Used by

The end-to-end suite spawns this sample as the agent side of its test scenarios, pairing it with the `samples/gguis/*` operator configurations (`default`, `canvas-demo`, `leaflet-demo`, `mapbox-demo`).

## Not used for

Not a published library. Not a scaffold. Not a framework. This file exists to be **read** by external developers building their own agents — the patterns here transfer directly to any MCP-host code path.
